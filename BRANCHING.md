# 分支策略

## 核心规则

1. **`main` 和 `AndroidUI` 永不合并。**
2. **版本进度保持同步。** 两个分支按相同的里程碑节奏推进，确保功能规划一致。
3. **平台差异各自消化。** 遇到某个平台不支持的功能（如网页端无法利用原生性能优化），直接略过，不强制降级替代。

## 分支说明

| 分支 | 路线 | 技术栈 | 说明 |
|------|------|--------|------|
| `main` | Web 版本 | Capacitor + HTML/CSS/JS | 主开发分支，CI 自动构建 APK |
| `AndroidUI` | 纯原生 Android | Kotlin/Jetpack Compose | 独立演进的原生版本 |

## 版本同步原则

- 两个分支共享同一份 **版本号体系**（如 v0.1、v0.2 …）
- 每个版本的功能清单（feature list）统一规划
- 实现时允许跳过不适用的功能：
  - Web 端跳过：原生性能优化、平台特有 API 等
  - AndroidUI 端跳过：WebView 特有的 CSS/H5 能力等
- 跳过 ≠ 偷懒，记录原因即可

## 为什么永不合并？

- 两条分支的技术栈完全不同（Web vs Native）
- 强行合并会产生大量冲突，且合并后的代码无法正常工作
- 两者共享的是**设计理念和功能规划**，不是代码

## 如果需要同步功能？

- 功能设计层面互相参考，保持进度对齐
- 具体实现必须各自独立编写
- 禁止 `git merge`、`git rebase`、`git cherry-pick` 跨分支操作
