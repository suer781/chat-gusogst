# 端点自适应重试与独立评分系统

> 设计文档

## 概述

当 API 请求失败时，系统自动在用户已配置的供应商列表中查找可替代端点，并利用 RAG 检索本地知识库推断端点路径格式。每个端点独立维护可靠性评分，互不干扰。

## 用户配置

每个供应商需要填写：

| 字段 | 必填 | 说明 |
|------|------|------|
| 域名 | 是 | 如 `api.deepseek.com` |
| API Key | 是 | 该供应商的密钥 |
| 端点路径 | 否 | 如 `/v1/chat/completions`，留空则自动检索 |

## 端点查找流程

```
用户填写域名（如 api.x.com）+ Key
    │
    ├── 有显式路径？ → 直接使用
    │
    └── 无显式路径？
          ↓
          RAG 检索本地知识库（内置供应商格式映射表）
          ↓
          找到语义最接近的端点路径
          ↓
          用此路径发起请求
```

## 内置知识库（纯本地，应用代码内置）

| 域名 | 推测路径 | 格式类型 |
|------|---------|---------|
| api.deepseek.com | /v1/chat/completions | OpenAI 兼容 |
| api.anthropic.com | /v1/messages | Anthropic |
| api.groq.com | /v1/chat/completions | OpenAI 兼容 |
| api.volcengine.com | /v1/chat/completions | OpenAI 兼容 |
| api.openai.com | /v1/chat/completions | OpenAI |
| ... | 更多条目随维护扩展 | ... |

## 独立评分公式（每端点各一份）

每个端点独立维护自己的评分、调用计数和错误计数：

```
k_m  = 12 / (m + 15)
C_m  = clamp( C_{m-1} · (1 - k_m) + ΔC,  -10000, 10000 )
```

| 符号 | 含义 |
|------|------|
| m | 该端点的调用次数（从 1 开始） |
| C_m | 该端点的当前评分，[-10000, 10000] |
| ΔC | 单次结果的评分变化量（成功正、失败负） |
| k_m | 学习率，随 m 增大而衰减 |

**端点评分完全隔离**：端点 A 的失败不会影响端点 B 的评分。

## 失败处理

- 前 3 次连续失败不计入错误计数（波动容忍）
- 第 4 次起错误计数 +1，ΔC 绝对值随计数增大
- 错误计数越大，降权灵敏度越高
- **成功一次后错误计数清零**

## 故障转移流程

```
主端点请求失败（超时/拒连/4xx/5xx）
    ↓
RAG 在供应商列表中检索格式最接近的端点
    ↓
找到候选？
    ├── 是 → 用该端点的 Key + 路径重试
    │        ↓
    │       成功？→ 继续，无感切换
    │       失败？→ 继续找下一个候选
    │
    └── 否 → 返回失败，提示用户
```

## 关键原则

1. **不串评分** — 每个端点独立计分，不存在「端点歧视」
2. **自带 Key 随切换** — 切到哪个端点就用哪个端点的 Key，不会跨供应商混用
3. **格式自适应** — RAG 基于路径格式语义检索，不依赖域名硬编码
4. **纯本地** — 知识库内置在代码中，向量检索本地执行，不依赖外部服务
