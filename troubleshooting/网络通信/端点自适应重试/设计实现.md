# 端点自适应重试系统 — 设计实现

## 概述

当 API 请求失败时，系统自动在用户已配置的供应商列表中查找可替代端点，并利用 EndpointKB（本地知识库）推断端点路径格式。每个端点独立维护可靠性评分，互不干扰。

## 核心组件

### 1. `EndpointScorer` — 独立评分引擎

每个端点（域名 + Key）有一个独立的 `EndpointScorer` 实例。

**公式：**
```
k_m = 12 / (m + 15)
C_m = clamp( C_{m-1} · (1 - k_m) + ΔC,  -10000, 10000 )
```

| 符号 | 含义 |
|------|------|
| m | 该端点的调用次数 |
| C_m | 当前评分，[-10000, 10000] |
| ΔC | 成功 +500，失败 -300 - 错误计数 × 150 |
| k_m | 学习率，随 m 增大衰减 |

**失败处理：**
- 前 3 次连续失败不计入错误计数（波动容忍）
- 第 4 次起错误计数 +1，每次失败惩罚递增 150
- 成功一次后错误计数清零

**健康判定：**
- `isHealthy()`: 评分 > 0 且（连续失败 ≤ 3 或错误计数 = 0）
- `shouldTryFallback()`: 连续失败 > 3 且评分 < 0

### 2. `EndpointKB` — 本地知识库

内置常见域名的 API 路径映射，纯本地运行。

**匹配策略（按优先级）：**
1. 精确域名匹配
2. 域名后缀匹配（如 `deepseek.com` 匹配 `api.deepseek.com`）
3. 域名主体匹配（去掉 `www.` / `api.` 前缀）
4. 标签关键词相似度
5. Fallback：默认 OpenAI 格式

**内置条目（2026-05-27）：**

| 域名 | 推荐路径 | 格式 |
|------|---------|------|
| api.openai.com | /v1/chat/completions | openai |
| api.deepseek.com | /v1/chat/completions | openai |
| api.anthropic.com | /v1/messages | anthropic |
| api.groq.com | /openai/v1/chat/completions | openai |
| dashscope.aliyuncs.com | /api/v1/chat/completions | openai |
| api.siliconflow.cn | /v1/chat/completions | openai |
| … | … | … |

（完整列表见 `EndpointKB.kt`）

### 3. `AutoRetryEngine` — 故障转移编排

1. 主端点失败 → `recordFailure()` 更新独立评分
2. `findFallbacks()` 从失败 URL 推测格式类型
3. 在用户已配置的供应商列表中找同格式、健康评分高的候选
4. 逐个尝试，成功者加分、失败者继续降权

### 4. ChatViewModel 集成

`callAiApi()` 修改要点：

- 调用前先用 `resolveEndpointPath()` 推断路径
- 流完成后记录 `recordSuccess()`
- HTTP/网络异常时记录 `recordFailure()`
- 主端点失败后轮询候选，消息中显示 `[自动切换: 供应商名 → 路径]`
- 全部失败则设置为 error 状态

## 用户可见行为

- 主端点工作正常时：**无变化**（完全透明）
- 主端点挂了有候选时：消息内容中出现 `[自动切换: xxx → /path]`
- 全部失败时：`所有可用端点均已失败，请检查网络或供应商配置`

## 后续注意事项

- EndpointKB 是硬编码列表，需随新供应商持续更新
- 评分是内存态，进程重启后重置。如需持久化需扩展 ChatStore
- 目前只匹配同格式（openai→openai），不支持跨格式转换
- tryEndpoint 中 `completed` 变量靠 var 闭包捕获，需注意线程安全性（当前单线程协程安全）
